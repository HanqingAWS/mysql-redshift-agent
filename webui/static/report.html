② 架构方案

客户端完全不感知，Proxy 在 SQL 层按表白名单分流；命中白名单的大表分析走 Redshift（经 LLM SQL 转换 + pgvector 知识库召回增强），未命中的小查询直连 MySQL。

三层知识叠加，越用越准：
① Markdown 规则前置（12 条硬编码正则 → 规则文档）处理方言差异的确定性部分， 零幻觉；
② pgvector 向量召回（历史成功样本 few-shot）覆盖 LLM 生僻场景和业务专有 SQL 形态；
③ LLM Bedrock Sonnet 4.6（global. 跨区推理）做最终 SQL 转换。
成功执行的 SQL 由 Proxy 异步回写到 pgvector，知识库随使用持续积累，翻译准确率单调上升。

⑤ 改造成本

以当前 demo 为蓝本，生产化落地一次性工作量 加起来只需要 1 天，月度运维 ≈ 0.5 人日。

一次性投入（合计 1 天）

  • Proxy 部署 （1 小时）
    Docker Compose 起 agent + proxy，配置 Redshift DSN / Bedrock Model ID
  • 白名单梳理 （1 小时）
    与业务方确认哪些表走 Redshift；目前 2–3 张订单大表
  • Redshift 建表 + DistKey/SortKey （1 小时）
    注意 TINYINT→INTEGER（Parquet 等宽）、DATETIME→TIMESTAMP
  • 同步脚本打磨 （2 小时）
    Snapshot → Parquet Export → COPY；KMS 授权、IAM Role
  • SQL 兼容性回归 （1 小时）
    收集历史慢 SQL 的典型样本批量跑一遍 Proxy
  • 灰度验证 （2 小时）
    并行跑 MySQL vs Proxy，校验结果一致性

持续成本

  • 月度同步 （0.5 人日/月）
    触发 06_planb_export.sh，跑完 COPY 校验 COUNT(*)
  • 知识库维护 （按需 / 0 人日）
    日常无需人工维护；Webui 可随时批量导入/删除样本
  • Bedrock Sonnet ≈ 数美元/月
    LRU Cache + 知识库召回，翻译几乎只发生在新 SQL 首次出现时
  • Cohere embedding < $1/月
    $0.0001/1K tokens，每天几千次翻译可忽略
  • Aurora PG Serverless ≈ $80/月
    MinACU=1 常驻（避免冷启动），突发自动扩到 18 ACU
  • Redshift Serverless 按秒计费
    8 RPU 空闲自动暂停；每月查询几千次 ≈ 十几美元
对比直接迁移：若逐条改写 SQL + 分析端切换，同体量项目历史上通常需要 6–10 人周；本方案把 80% 的工作量消化在 Proxy + LLM 中，人工只负责 20% 的边界情况，一次性投入从数周压缩到 1 天。

⑥ 知识库与向量召回

在规则前置之上叠加一层 pgvector 向量召回，把"历史验证过的 MySQL/Redshift 成功对"作为 few-shot 动态贴到 prompt。用得越多，覆盖越广，翻译越稳。

📥 知识如何进库

  • runtime 自动沉淀：Proxy 每次成功执行的 (mysql_sql, redshift_sql) 通过 POST /save_example 异步回写（fire-and-forget，不阻塞客户端）
  • import 批量导入：Webui 上传 SQL 文件 → 每条走"翻译 + MySQL 执行 + Redshift 执行 + 结果对比"，通过才入库；对比失败可人工强制入库（标记 override）
  • seed 冷启动：scripts/seed_knowledge.py 用 6 条已验证样本预热
  • 去重：UNIQUE(md5(mysql_sql)) 索引 + ON CONFLICT ... DO UPDATE upsert，相同 SQL 只存一条，hit_count 自增

🔎 如何被召回

  • Embedding：cohere.embed-multilingual-v3（1024 维，中英混排鲁棒）
  • 索引：pgvector HNSW (vector_cosine_ops, m=16, ef=64)，50 万条级别 依然单次毫秒返回
  • 触发：Agent 在调 Bedrock 之前，embed(sql, search_query) → 查 top-3 → 相似度 ≥0.85 的贴进 prompt
  • 降级：Aurora PG 不可用 → 回落到"只用规则 + LLM"，客户端无感
召回实测（同一个 IFNULL 变体改值）
#查询 SQLexamples_hit相似度翻译结果
种子 SELECT uid, IFNULL(app_version,'unknown') FROM dw.ads...iap_data_new LIMIT 3 —— 入库 source=seed
变体 SELECT uid, IFNULL(app_version,'n/a') AS ver FROM dw.ads...iap_data_new LIMIT 5 10.90 命中召回 → 转换正确
不相干 SELECT COUNT(*) FROM users_profile WHERE region='jp' 0— 阈值过滤，走纯 LLM
两个现实效果： (1) 客户业务常跑的 SQL 只需要人工校验一次，以后同类形态都能秒翻；
(2) LLM 对生僻或客户专有的 SQL 模式（例如罕见 JOIN 顺序、内部函数命名）有了 稳定的 few-shot 参考，不再靠"运气"。
Aurora PG Serverless v2： MinACU=1, MaxACU=18。MinACU=1 避免冷启动（保证首次翻译毫秒级查向量库）， MaxACU=18 满足批量导入时几十条并发 embedding 的突发压力；闲时按 1 ACU 计费， 月度约 $80。

⑧ 结论与建议

✅ 适合采纳此方案的场景

  • 客户端代码是"黑盒"或团队成本高（SAP/ERP/老 PHP/闭源 BI）
  • 热数据与冷数据边界清晰，分析表集合稳定
  • 可接受数据 T+N 新鲜度（月/日/小时同步）
  • OLTP 负载仍需留在 MySQL（订单写入、会员查询）
  • 分析类 SQL 形态有限、可缓存复用

⚠️ 不适合 / 需重新评估的场景

  • 跨库 JOIN（订单表 JOIN 用户表，两张在不同引擎）
  • 秒级实时要求（应直接走 CDC 流式方案）
  • 高频 SQL Variance（每条都是独特的，缓存命中率极低 → Bedrock 成本上升）
  • 严格合规场景（SQL 经第三方 LLM 往返需审批）
推荐落地顺序：
① 用当前 2 张订单表做为期 1 个月的影子模式（双写 Proxy + 直连 MySQL，比对结果）
② 第 2 个月切流量到 Proxy，保留 MySQL 做兜底
③ 第 3 个月稳定运行后，扩展白名单到第 2 批分析表
